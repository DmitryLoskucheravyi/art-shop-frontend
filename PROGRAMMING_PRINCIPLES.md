# Programming Principles

## 1. DRY (Don't Repeat Yourself)

У проєкті частина логіки винесена в окремі функції, щоб не дублювати однаковий код у різних місцях.

Наприклад, для роботи з HTTP-запитами використовується окрема функція `fetchJson(url, options = {})`, яка інкапсулює:
- виклик `fetch`
- встановлення `Content-Type`
- перевірку `response.ok`
- перевірку типу відповіді
- перетворення у JSON

Це дозволяє не повторювати одну й ту саму логіку в `addToBasket()`, `getBasketItems()` та `getArtCards()`.

Приклади в коді:
- `fetchJson(...)`
- `addToBasket(...)`
- `getBasketItems()`
- `getArtCards(...)`

Також для перевірки коректності товару використовується окрема функція `isValidArtItem(item)`, щоб не дублювати перевірки полів у різних місцях.

---

## 2. Separation of Concerns

У коді простежується часткове розділення відповідальностей між різними частинами логіки.

Наприклад:
- клас `Art` відповідає за створення та рендер картки товару
- `addToBasket()` відповідає за додавання товару в кошик
- `getBasketItems()` відповідає за отримання товарів із сервера
- `deleteItemOfBasket()` і `deleteItemOfBasketAll()` відповідають за видалення товарів з кошика
- `swapLang(currentLang)` відповідає за перемикання мови інтерфейсу
- `BasketRender()` відповідає за оновлення вмісту кошика

Такий підхід робить код зрозумілішим, оскільки різні задачі не зведені повністю в одну велику функцію.

---

## 3. Encapsulation

У класі `Art` дані про товар та логіка його відображення згруповані в одному місці.

Конструктор класу приймає властивості товару:
- `id`
- `title`
- `artSize`
- `paintType`
- `paintTypeUa`
- `year`
- `img`
- `priceUSD`
- `priceUAH`

Метод `render()` використовує ці дані для створення DOM-елемента картки товару та прив’язки обробника події для кнопки покупки.

Це є прикладом інкапсуляції, оскільки стан об’єкта та пов’язана з ним поведінка об’єднані в межах одного класу.

---

## 4. Defensive Programming

У проєкті є елементи захисного програмування, які зменшують ризик помилок під час виконання.

Приклади:
- перевірка існування `selectLang` перед додаванням обробника події
- перевірка валідності об’єкта товару через `isValidArtItem(item)`
- використання `try...catch` у функціях роботи з сервером
- перевірка, чи відповідь сервера є масивом: `Array.isArray(data)`
- очищення `basketMainDb` і повторний рендер кошика у випадку помилки завантаження

Це робить застосунок стабільнішим при неочікуваних даних або помилках сервера.

---

## 5. Readability and Maintainability

Для підвищення читабельності в коді використовуються:
- зрозумілі назви багатьох змінних і функцій
- попереднє збереження DOM-елементів у константи
- логічне групування функцій за призначенням
- асинхронні функції для окремих операцій із сервером

Наприклад, такі назви як:
- `getBasketItems`
- `deleteItemOfBasket`
- `deleteItemOfBasketAll`
- `swapLang`
- `basketItemContainer`
- `basketTotalTable`

дозволяють швидше зрозуміти призначення коду.
